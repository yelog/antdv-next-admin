# 复杂表单进度反馈设计

## 现状

复杂表单已经通过 `ProStepForm` 使用 `a-steps` 展示“基本信息、策略配置、发布设置”三个阶段。现有实现能够指出当前步骤，但采用默认小尺寸样式，缺少总体完成度和当前阶段说明，在信息密度较高的表单中不够醒目。

## 设计目标

- 同时回答“当前在哪里”“还剩多少”“当前要完成什么”。
- 保留步骤导航的阶段语义，不改变现有表单校验、草稿和提交行为。
- 桌面端完整呈现步骤关系；窄屏减少重复信息，避免步骤标题挤压。
- 能力沉淀到通用 `ProStepForm`，业务页面只负责提供步骤文案。

## 方案

在现有步骤条上方增加进度摘要，包含“第 N / M 步”、百分比和当前步骤说明；摘要下方使用一条紧凑线性进度条。步骤进度按当前阶段计算：第一步为 `1 / M`，最后一步为 `M / M`。

桌面端继续展示完整 `a-steps`，为每一步补充一句结果导向的说明。小于 600px 时隐藏完整步骤条，仅保留进度摘要与线性进度条，让窄屏仍能清楚感知流程位置而不产生纵向步骤条带来的额外高度。

## 视觉层级

1. 进度摘要使用中等字重，成为表单标题后的第一层流程信息。
2. 当前步骤说明使用次级文字色，不与字段标签争夺注意力。
3. 线性进度条使用主题主色和圆角轨道，不引入新的强调色。
4. 完整步骤条作为阶段地图，和进度摘要保持适度间距。

## 可访问性与边界

- 进度摘要使用 `aria-live="polite"`，步骤变化时可被辅助技术感知。
- 空步骤数组时进度为 0，避免除零和非法索引。
- 外部传入越界步骤时对展示值做边界收敛，不改变既有双向绑定行为。
- 进度组件关闭自带数值，避免和摘要中的百分比重复播报。

## 验证

- 对进度计算补充单元测试，覆盖首步、中间步、末步、空数组和越界输入。
- 运行目标单测、类型检查、lint 和生产构建。
- 在桌面宽度和窄屏宽度验证信息层级、步骤切换和按钮行为。
