# 菜单搜索最佳匹配排序设计

## 背景

全局菜单搜索当前只判断标题、路径或拼音是否命中，然后保留菜单树的遍历顺序。结果顺序因此取决于路由配置，而不是用户输入与菜单名称的相关度。

## 架构方案

将匹配与排序从 `GlobalSearch.vue` 提取为纯 TypeScript 领域函数。Vue 组件继续负责菜单树展平、交互和渲染；搜索模块负责文本归一化、候选字段匹配、质量比较、稳定排序和数量限制。

每个菜单提供三个候选字段：叶子标题、完整面包屑标题和路由路径。字段优先级依次降低，避免路径中的偶然命中压过菜单名称。

匹配质量使用结构化元组比较，不使用难以解释的魔法总分：

1. 字段优先级：叶子标题、完整标题、路径。
2. 匹配类型：完全匹配、前缀匹配、连续子串、非连续拼音或字符匹配。
3. 连续度：最长连续段更长者优先。
4. 紧凑度：间隔更少、总跨度更短者优先。
5. 位置和长度：更早命中、更短字段优先。
6. 原始顺序：所有质量相同时保持菜单树顺序，保证结果稳定。

## 边界

- 继续使用现有 `pinyin-pro`，不引入新的模糊搜索依赖。
- 保持现有大小写不敏感、拼音匹配和最多 20 条结果的行为。
- 不加入访问历史、收藏或业务权重，避免相关度和个性化规则耦合。
- 空白输入不产生结果；输入会先去除首尾空白。

## 验证策略

使用纯函数单元测试覆盖连续匹配、字段优先级、前缀和位置、拼音匹配、稳定排序以及排序后截断。组件仅保留一次集成调用，通过类型检查和生产构建验证接线。
